# Support

Use this document to choose the right public support path for Kern.

## Helpful docs before filing

- Install a published binary: [Installing Kern from a GitHub release](docs/release/installing-kern-from-github-release.md)
- Build from source: [Building Kern from source](docs/release/building-kern-from-source.md)
- Current public release gate: [Release validation gate](docs/release/release-validation-gate.md)

## Bug reports

For reproducible bugs in Kern, open a GitHub issue with the bug-report form.

Use issues for:
- editor bugs
- rendering/import/export regressions
- packaging/install bugs for published release assets when the issue is ordinary install friction
- documentation problems

Use the private security reporting path in `SECURITY.md` instead of a public issue if you see:
- a checksum mismatch for a published release asset
- suspicious DMG or app bundle contents
- a release asset that appears tampered with or replaced

## Feature requests

For new features or larger behavior changes, open a GitHub issue with the feature-request form.

## Security issues

Do **not** open a public issue for an undisclosed security problem.

Use the private vulnerability reporting path described in `SECURITY.md`.

## Code of conduct / abuse concerns

For project conduct concerns, see `CODE_OF_CONDUCT.md`.

If the problem involves harassment, abuse, or another GitHub-platform concern that needs GitHub intervention, use GitHub Support's private contact path.

## What this repository does not provide

This repository does not currently provide:
- private general support
- paid support
- guaranteed response times for non-security issues

## Before opening an issue

Please include:
- macOS version
- Kern revision or release tag
- the smallest reproduction you can provide
- relevant validation you already ran
